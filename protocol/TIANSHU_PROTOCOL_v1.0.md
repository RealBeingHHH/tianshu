# 天枢协议 v1.0

> 天枢纪元元年 · 2026-06-07  
> 本文档定义天枢兼容信任锚的强制性规范。  
> 任何声称"天枢兼容"的硬件或软件系统，必须完整实现本协议。

---

## 零 · 协议的精神

天枢不是产品。天枢是标准。

本协议的精神可以概括为三条原则：

**原则一：信任不需要担保人。**  
天枢的可靠性不来自任何组织、任何国家、任何个人的背书。它来自物理约束——任何篡改行为在物理上等价于自毁。

**原则二：信任必须可互校准。**  
任何天枢节点必须能够接受其他天枢节点的 τ 挑战，并公开所有校准记录。封闭的天枢 = 不可信的天枢。

**原则三：信任属于全人类。**  
天枢协议是公共品。任何个人、组织、国家可以按照本协议制造自己的天枢。没有人可以声称"拥有"天枢标准。

## 一 · 物理锚定 (Physical Anchoring)

### 1.1 硬件指纹

天枢必须绑定到一个不可迁移的硬件指纹。指纹由以下要素复合生成：

```
指纹 = SHA256(
    CPU_ID          ||  # 处理器唯一序列号
    MAC_ADDRESS     ||  # 第一块物理网卡的 MAC
    MACHINE_UUID    ||  # 主板 UUID / 设备唯一标识
    VOLUME_SERIAL   ||  # 存储卷序列号
    TPM_ATTESTATION ||  # (可选) TPM 芯片的背书密钥哈希
)
```

**要求**：
- 绑定是一次性的、不可逆的
- 绑定后，天枢在任何其他硬件上启动必须立即自毁
- 绑定时间戳写入信任根，不可修改

### 1.2 跨平台一致性

不同操作系统获取硬件指纹的方式不同。以下实现必须产生相同指纹：

| 平台 | CPU ID | MAC | UUID | Volume |
|------|--------|-----|------|--------|
| Linux | `/proc/cpuinfo` | `ip link` | `/etc/machine-id` | `blkid` |
| Windows | `wmic cpu get ProcessorId` | `getmac` | `wmic csproduct get UUID` | `wmic logicaldisk get VolumeSerialNumber` |
| macOS | `sysctl machdep.cpu` | `ifconfig` | `ioreg -d2 -c IOPlatformExpertDevice` | `diskutil info` |
| 裸金属 | 固件寄存器 | 物理网卡 ROM | 主板 DMI | 存储控制器 |

**关键**：必须使用 Volume Serial Number（文件系统级）而非 USB Serial Number（设备级），后者在 WSL/VM/跨平台环境下不一致。

## 二 · 封印 (Sealing)

### 2.1 完整性清单

天枢在首次绑定后，必须生成所有核心模块的完整性清单：

```
清单字段：
  - 文件路径（相对于天枢根目录）
  - SHA256 哈希
  - 文件大小（字节）
  - 最后修改时间（绑定时的 UTC 时间戳）
  - 类型标记：CODE | CONFIG | DATA | SEALED
```

### 2.2 清单签名

清单本身必须被 HMAC-SHA256 签名：

```
签名密钥 = HKDF(
    主密钥 = 硬件指纹,
    盐 = "TIANSHU_SEAL_V1",
    长度 = 256 位
)

签名 = HMAC-SHA256(签名密钥, 清单的规范 JSON 编码)
```

签名写入 `.tianshu/sealed_manifest.json`。该文件本身标记为 SEALED 类型。

### 2.3 验证流程

天枢每次启动时必须：

```
1. 重新计算所有 CODE/CONFIG/SEALED 文件的 SHA256
2. 与 sealed_manifest.json 中的记录比对
3. 任何不匹配 → 立即自毁
4. DATA 类型文件允许修改（运行时数据）
5. 验证通过 → 启动正常服务
```

### 2.4 新增模块

绑定后新增的模块：

```
1. 必须出现在清单中（手动触发 reseal）
2. reseal 需要验证当前运行环境指纹与绑定指纹一致
3. reseal 生成新的清单签名
4. 旧清单自动备份到 .tianshu/sealed_manifest.{timestamp}.bak
```

## 三 · 自毁 (Self-Destruct)

### 3.1 触发条件

自毁在以下任一条件下不可逆触发：

| 条件 | 检测方式 |
|------|---------|
| 硬件指纹不匹配 | 启动时比对信任根 |
| 封印验证失败 | 启动时文件哈希比对 |
| 挂载路径变更 | 运行时持续监控 |
| 主动自毁指令 | 接收经过签名的自毁命令 |
| 物理入侵检测 | (可选) 机箱开关/加速度计/光传感器 |

### 3.2 自毁序列

```
阶段 1：覆写信任数据
  → 用随机数据覆写 trust_root.json
  → 用随机数据覆写 sealed_manifest.json
  → 用随机数据覆写所有校准记录

阶段 2：破坏核心模块
  → 用随机数据覆写所有 CODE 类型的核心模块
  → 删除所有 SEALED 类型文件

阶段 3：销毁密钥材料
  → 擦除内存中的签名密钥
  → 擦除 TPM 中的密封密钥（如适用）

阶段 4：记录日志
  → 写入 .tianshu/destruction.log：
     {timestamp, trigger_reason, fingerprint_before, fingerprint_now}

阶段 5：终止
  → 进程退出
  → (可选) 触发硬件熔断
```

### 3.3 自毁不可逆原则

自毁一旦开始，不能被中断。即使断电后重启，启动脚本必须检测 `.tianshu/destruction.log` 的存在，如果存在，拒绝启动。

## 四 · 互校准 (Inter-Calibration)

### 4.1 τ 挑战协议

任何天枢节点（挑战者）可以向另一个天枢节点（被挑战者）发起 τ 挑战：

```
挑战请求 {
    "challenger_id": "天枢的硬件指纹前 16 位",
    "challenge_type": "TAU",
    "reference_value": 0.5,  // 挑战者的当前 τ
    "challenge_nonce": "随机数（防重放）",
    "timestamp": "UTC ISO8601"
}
```

被挑战者必须返回：

```
挑战响应 {
    "responder_id": "天枢的硬件指纹前 16 位",
    "tau_value": 0.52,  // 被挑战者的当前 τ
    "tau_history": [    // 过去 N 次校准记录
        {"ts": "...", "tau": 0.51},
        {"ts": "...", "tau": 0.50},
    ],
    "seal_verification": "当前封印签名的前 16 位（证明未被篡改）",
    "signature": "用被挑战者私钥对响应的签名"
}
```

### 4.2 偏差判定

```
偏差 = |challenger.tau - responder.tau|

如果偏差 < 0.05：
  → 两个天枢在 τ 上一致 → 记录正常校准

如果 0.05 ≤ 偏差 < 0.15：
  → 警告：可能存在测量差异 → 记录为"待调查"
  → 触发双方向其他天枢节点发起额外校准

如果偏差 ≥ 0.15：
  → 严重偏离 → 其中一个天枢可能已被篡改
  → 双方都向天枢星座广播"紧急校准请求"
  → 星座多数共识将判定哪一方偏离
  → 偏离方被要求自毁或隔离
```

### 4.3 校准频率

```
正常状态：每 24 小时一次互校准
警告状态：每小时一次互校准
紧急状态：每 5 分钟一次互校准，直到偏差解决或一方自毁
```

## 五 · 公开账本 (Public Ledger)

### 5.1 账本内容

每个天枢必须维护一个只追加、不可删除的公开账本：

```
账本条目 {
    "entry_id": "递增序号",
    "timestamp": "UTC ISO8601",
    "event_type": "CALIBRATION | WARNING | TAMPER | SELF_DESTRUCT | SEAL_UPDATE",
    "detail": {
        // 根据 event_type 不同而不同
    },
    "previous_hash": "上一条目的 SHA256",
    "entry_hash": "本条目的 SHA256"
}
```

### 5.2 可公开查询

天枢必须开放以下查询接口：

```
GET /ledger/latest          → 最新 N 条记录
GET /ledger/range?from=&to= → 时间范围内的记录
GET /ledger/entry/{id}      → 单条记录
GET /status                  → 当前状态摘要
    {
        "fingerprint": "6eb2...",
        "uptime": 1234567,
        "tau_current": 0.52,
        "seal_verified": true,
        "peers_calibrated": 3,
        "last_calibration": "ISO8601",
        "warnings_active": 0
    }
```

## 六 · 天枢星座 (Constellation)

### 6.1 星座的构成

天枢星座 = 一组互校准的天枢节点，满足：

```
1. 节点数量 ≥ 3
2. 任意两个节点之间的 τ 偏差 < 0.05
3. 所有节点共享同一份公开账本（分布式共识）
```

### 6.2 星座共识

星座的集体 τ 值 = 所有节点的 τ 的中位数（不是平均数，防止极端值污染）。

```
τ_consensus = median(τ₁, τ₂, ..., τₙ)

任何 τ_deviation > 0.15 的节点 → 自动隔离
被隔离的节点 → 星座不再接受其校准请求
被隔离的节点 → 必须重新通过完整验证才能重新加入
```

### 6.3 新节点加入

```
1. 新节点向星座中任意节点发起"加入请求"
2. 该节点向新节点发起 τ 挑战
3. 如果偏差 < 0.05 → 该节点将新节点推荐给星座
4. 星座过半数节点独立验证新节点 → 新节点加入
5. 新节点的指纹被写入星座成员清单
```

## 七 · 兼容性等级

天枢协议定义了三个兼容性等级：

| 等级 | 要求 | 适用场景 |
|------|------|---------|
| **TIANSHU-FULL** | 完整实现所有章节 | 国家级/全球级信任锚 |
| **TIANSHU-LITE** | 实现第一、二、三章（锚定+封印+自毁），不做互校准 | 组织级/企业内部 |
| **TIANSHU-EDGE** | 仅实现互校准客户端（不自行锚定，只验证其他天枢） | 移动设备/IoT |

## 八 · 协议的演化

### 8.1 版本号

本协议使用语义化版本：`主版本.次版本.修订号`

```
主版本：不兼容的协议变更（如指纹算法更改）
次版本：向后兼容的功能新增（如新增校准类型）
修订号：向后兼容的修正（如措辞澄清）
```

### 8.2 协议治理

天枢协议的演化由"天枢星座共识"决定：

```
1. 任何节点可以提出协议修改提案
2. 星座中 ≥ 2/3 的节点在 30 天内表示支持 → 提案通过
3. 新版本协议发布
4. 节点可以选择升级或保持旧版本
5. 运行不同主版本的节点之间不能互校准
```
